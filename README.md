# FinTrack

Personal finance tracking for Android: expenses and income, loans, and utility
bills. Works without a server — all your data stays on the phone.

## Why

Finance apps usually either make you create an account or calculate loan
interest approximately. FinTrack does neither:

- **No server.** No sign-up, no sync, nothing to leak.
- **Money in kopecks.** Every amount is a whole `Long`. Not a single `Double`
  in the calculations.
- **Bank-accurate interest.** Balance × rate / 365 (366 in a leap year) × the
  actual number of days in the period, rounded HALF_UP on every payment. The
  formulas are verified against a real loan agreement down to the kopeck.

## Features

**Transactions.** Three-tap quick entry, categories with icons, swipe to edit
and delete, statistics by category and by month.

**Loans.** Annuity, differentiated, installment, and interest-only. The payment
schedule is not stored in the database — it is generated by a pure function, so
a rate change or an early repayment simply recomputes the whole thing, leaving
no room for it to drift out of sync. The early-repayment simulator compares
"shorten the term" against "lower the payment".

**Obligatory payments.** Subscriptions and recurring bills sit in one list
together with loans. Reminders fire on the due date; marking a payment "paid"
creates a real transaction — the balance reflects what actually happened, not
what was planned.

**Utilities.** Meters with tariffs, reading entry with automatic calculation,
services billed by norm or as a fixed amount. Grouped by bill, with batch
reading entry and paying several services at once. Past readings keep their own
tariff, so changing a tariff does not rewrite history.

**Widgets.** Balance showing the money left after obligatory payments, plus a
list of upcoming payments. Three sizes, with contents that adapt.

## Installation

The APK is in the [releases](https://github.com/Sch1z0eD/FinTrack/releases).
Android will ask for permission to install from this source.

The app can check for updates itself: **Settings → About**. The check is off by
default and only runs on a button press until it is turned on — this is the only
time the app goes online.

## Building

Requires JDK 21.

```bash
./gradlew assembleDebug        # build
./gradlew installDebug         # install onto a connected phone
./gradlew :loanengine:test     # loan engine tests
./gradlew lint :loanengine:test  # before committing
```

The three build types are three separate apps that sit on the phone at the same
time without getting in each other's way:

| type | app | updates from |
|---|---|---|
| `debug` | `com.findev.fintrack.debug` — "FinTrack dev" | nothing, installed from Android Studio |
| `beta` | `com.findev.fintrack.beta` — "FinTrack beta" | beta releases on GitHub |
| `release` | `com.findev.fintrack` — "FinTrack" | stable releases |

It is done this way because Android will not let you replace an app with a build
signed by a different key: without the split, every switch between debug and
release would mean reinstalling and losing the database.

## How it is built

```
:app          Compose UI, Room, Hilt, WorkManager, widgets
:loanengine   loan math — pure Kotlin, no Android
```

Kotlin, Jetpack Compose, Material 3, Room, Hilt, WorkManager, Glance, Vico.
minSdk 31.

The loan engine lives in a separate module with no dependency on Android at all:
the payment schedule is arithmetic, and it needs to be checkable with unit tests
rather than on an emulator. The oracles in the tests are computed by hand from
real loan agreements, not generated by the same engine — otherwise the test
would be checking itself.

The screens follow MVVM: Screen → ViewModel → Repository → DAO. The ViewModel
knows nothing about Compose, and composables never reach into repositories.

## Releases

The channel is set by the tag, not the branch:

```bash
git tag v1.2.3      && git push origin v1.2.3       # stable release
git tag v1.2.4-beta && git push origin v1.2.4-beta  # prerelease, beta app only
```

`versionCode` and `versionName` are derived from the tag, so a published build
cannot disagree with its number. The channels share one version sequence that
simply keeps growing — betas and stable versions never meet, because they are
different apps.

The stable app reads `/releases/latest`, which by GitHub's own definition never
returns a prerelease — so betas cannot reach it even by accident. Work happens
on the `dev` branch; only what is ready for release lands in `master`.
